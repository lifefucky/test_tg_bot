# Планы и проектные заметки

Сюда складывают **черновики планов доработок**, обзоры задач и решений, которые не должны дублировать «истину» в коде или в [`docs/`](../docs/).

Примеры:

- планы, сгенерированные в Cursor (файлы вида `*.plan.md`);
- RFC по новой фиче до переноса в `docs/architecture.md` после реализации.

**Правило:** после внедрения фичи актуальное описание — в [`docs/architecture.md`](../docs/architecture.md) и профильных файлах [`docs/`](../docs/); здесь можно оставить ссылку на PR или краткую историю.

### Наименование

Логирование запросов к API onlinecontract (`fetch_modules/online_trade.py`)

### Проблема

При ошибках и нестандартных ответах API (в т.ч. `status_code != 200`, исключения `requests`) **сложно отлаживать инциденты** и отличать сценарии: в логе нет единообразной фиксации кода ответа, контекста и типа ситуации; приходится полагаться на сырой вывод или догадки.

### Направление

Добавить **логирование** с обязательными элементами:

- **HTTP-код ответа** — всегда фиксировать в логе.
- **Фрагмент тела ответа** — ограниченная длина (например, первые N символов), чтобы не заливать логи и не тащить в лог лишнее.
- **Человекочитаемый комментарий / категория ситуации** — например: «процедура недоступна», «временная недоступность сервиса», «ошибка авторизации», «некорректный ответ» и т.п. (маппинг по коду/типу ошибки или по содержимому при реализации).

Цель — проще отлаживать инциденты **без** обязательного вывода сырого кода в Telegram пользователю.

### Наименование

Выбор категорий в более «нативном» формате

### Проблема

В источнике (площадка / API) задано **очень много категорий и подкатегорий**; линейный или плоский выбор в боте ведёт к тому, что пользователь **теряется** в длинных списках и не находит нужное.

### Направление

Дать пользователю выбор категорий в формате, более привычном для Telegram и мобильного UX: пошаговый выбор (дерево), поиск/фильтр по названию, избранное, «недавние», сокращение первого экрана до нескольких понятных групп с углублением дальше. Конкретная реализация (inline-кнопки, команды, mini-app) — на этапе проектирования.

### Наименование

Расширенное взаимодействие с платформой OnlineContract

### Проблема

Текущий сценарий бота **узкий**: по сути список процедур по категориям и конкурсный лист; возможности платформы OnlineContract **использованы не полностью**, продуктовый потолок по интеграции не задан.

### Направление

Выйти за рамки текущего сценария и **глубже использовать возможности платформы** — детали, приоритеты и объём зафиксировать отдельно после анализа API и продуктовых целей.
