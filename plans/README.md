# Планы и проектные заметки

Сюда складывают **черновики планов доработок**, обзоры задач и решений, которые не должны дублировать «истину» в коде или в [`docs/`](../docs/).

Примеры:

- планы, сгенерированные в Cursor (файлы вида `*.plan.md`);
- RFC по новой фиче до переноса в `docs/architecture.md` после реализации.

**Правило:** после внедрения фичи актуальное описание — в [`docs/architecture.md`](../docs/architecture.md) и профильных файлах [`docs/`](../docs/); здесь можно оставить ссылку на PR или краткую историю.

## Логирование запросов к API onlinecontract (`fetch_modules/online_trade.py`)

Нужно добавить **логирование** при ошибках и нестандартных ответах (в т.ч. при `status_code != 200` и при исключениях `requests`):

- **HTTP-код ответа** — всегда фиксировать в логе.
- **Фрагмент тела ответа** — ограниченная длина (например, первые N символов), чтобы не заливать логи и не тащить в лог лишнее.
- **Человекочитаемый комментарий / категория ситуации** — например: «процедура недоступна», «временная недоступность сервиса», «ошибка авторизации», «некорректный ответ» и т.п. (маппинг по коду/типу ошибки или по содержимому, как решите при реализации).

Цель: проще отлаживать инциденты и отличать сценарии без обязательного вывода сырого кода в Telegram пользователю.
